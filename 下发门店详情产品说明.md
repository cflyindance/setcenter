# 下发门店详情 — 产品说明（文档辅助）

> **实现位置**：`kiosk-theme-list.html` 内模态层 `#storesModal`、移除确认 `#storeRemoveConfirmModal` 及相关脚本。  
> **数据归属**：当前主题 `theme.distributedStores`（与 `kiosk_themes` 中该条主题一并持久化在 `localStorage`）。  
> **上游写入**：门店列表在完成下发流程时由 `effective-time.html` 生成（含模拟下发结果）；与流程衔接见 [`屏保新建到完成流程说明.md`](屏保新建到完成流程说明.md)。  
> **状态语义补充**：主题级下发/生效聚合见 [`屏保状态与操作说明.md`](屏保状态与操作说明.md)。

---

## 1. 产品定位与目标

| 项目 | 说明 |
|------|------|
| **是什么** | 在「屏保主题列表」中，针对**已执行过下发**的主题，查看本次下发涉及的**门店清单**及每家门店的**门店维度结果**、**分渠道结果**，并在规则允许时进行**单店运营操作**（下架 / 重新上架 / 从记录中移除）。 |
| **不是什么** | 不是重新选择门店的下发向导（那是 `store-select.html`）；**不支持**在弹窗内修改门店名称或 MID；**不根据**门店上下架状态重算主题级「下发状态」文案（主题级仍以门店 `status` 聚合为准）。 |
| **典型用户** | 运营 / 配置人员：核对下发结果、处理失败重试、按店控制展示在架与记录清理。 |

---

## 2. 入口与打开条件

| 条件 | 行为 |
|------|------|
| 列表卡片上存在 **「查看详情」** 按钮 | 仅当该主题已有 `distributedStores` 且 `length > 0` 时展示（无下发记录则无入口）。 |
| 用户点击 **查看详情** | 调用 `openStoresModal(themeId)`。 |
| 主题存在且 `distributedStores` 非空 | 打开弹窗：写入 `currentModalThemeId`，重置筛选项（下发状态 = 全部、MID 为空），统计成功/下发中/失败数量，决定是否展示 **重试失败下发**，再渲染列表。 |
| 主题不存在或无 `distributedStores` | **不打开**弹窗（无提示，静默）。 |

说明：当前实现**不**因主题处于「逻辑已生效且未屏保下架」而禁止打开本弹窗（可查看详情并做允许范围内的门店操作）。

## 3. 界面结构（信息架构）

### 3.1 顶栏

| 元素 | 说明 |
|------|------|
| 标题 | 「下发门店详情」 |
| 关闭 | 「×」；点击遮罩空白处关闭；`Esc` 关闭（若移除确认层已打开则先关确认层）。 |

### 3.2 统计条（全量，不受筛选影响）

| 指标 | 计算逻辑 |
|------|----------|
| **总计** | `distributedStores.length` |
| **成功** | `status === 'success'` 的条数 |
| **下发中** | `status === 'pending'` 的条数 |
| **失败** | `status === 'failed'` 的条数 |

每次打开弹窗或刷新弹窗内数据时按**当前主题全部门店**重算；与下方列表筛选**无关**。

### 3.3 「重试失败下发」按钮

| 显示条件（满足其一即可） | 判断逻辑 |
|--------------------------|----------|
| 存在失败门店 | `failedCount > 0` |
| 存在渠道级失败 | `hasChannelLevelFailure(stores)`：至少一家门店的 `channelStatuses` 中存在 `status === 'failed'` |

否则按钮 **隐藏**。  
点击后使用浏览器 **`confirm`** 二次确认，确认后执行重试逻辑（见 §6）。

### 3.4 筛选与提示区

| 控件 | 行为 |
|------|------|
| **下发状态** 下拉 | 选项：全部 / 成功 / 下发中 / 失败；变更时重绘列表（仅过滤展示，不改数据）。 |
| **MID** 输入框 | 实时按子串、**不区分大小写**匹配 `store.mid`；空则不过滤 MID。 |
| **重置** | 清空状态筛选与 MID，重绘列表。 |
| 筛选结果提示 `#storesFilteredHint` | 当有筛选条件或结果条数小于总数时显示：`当前显示 x / y 家门店`。 |
| 黄色说明文案 | 固定提示：**成功门店**支持单个下架/重新上架；**下架后**可从记录中删除；**不支持**在此修改门店名称或 MID。 |

### 3.5 门店列表（每条卡片）

| 区域 | 内容 |
|------|------|
| 左侧主信息 | 门店名称、`MID:`；成功且有 `successTime` 时展示「下发成功: &lt;时间&gt;」；失败且有 `failedTime` 时展示「下发失败: &lt;时间&gt;」；失败且无 `failedTime`（旧数据）时展示「下发失败: —」 |
| 右上徽标 | **门店：** + 门店维度状态（成功 / 下发中 / 失败），颜色与列表内约定一致 |
| **各渠道下发** | 当 `channelStatuses` 存在且非空时展示；每行：渠道名 + 该渠道状态（成功/下发中/失败等） |
| **上下架与删除** | 仅当 **门店维度为成功** 时渲染（见 §5） |

无匹配筛选结果时展示空态文案：「无匹配门店，请调整下发状态或 MID 关键词」。

## 4. 数据模型（单条 `distributedStores[i]`）

以下为原型中与弹窗强相关的字段（非穷尽主题上所有字段）。

| 字段 | 类型（概念） | 含义 |
|------|--------------|------|
| `name` | 字符串 | 门店名称（展示，不可在弹窗内编辑） |
| `mid` | 字符串 | 门店 MID；操作中作为**行内唯一键**匹配（字符串比较） |
| `status` | `success` \| `pending` \| `failed` | **门店维度**下发结果 |
| `successTime` | 字符串（可选） | 门店维度为成功时的展示时间（完成下发或重试成功时写入） |
| `failedTime` | 字符串（可选） | 门店维度为失败时的展示时间（完成下发或重试仍为失败时写入；成功后会清除） |
| `channelStatuses` | `{ channel, status }[]`（可选） | 各渠道在该门店上的下发结果；`status` 取值与门店级类似，用于展示与「渠道级失败」判断 |
| `shelfStatus` | `'off_shelf'` 或缺省 | **仅对成功门店有意义**：缺省或非 `off_shelf` 视为**在架**；`off_shelf` 为**已下架** |

写入时机概要：完成下发时在 `effective-time.html` 内按场景随机生成门店与渠道状态；成功项可带 `shelfStatus: 'on_shelf'`（与「在架」展示一致）；重试成功时若未处于下架则置为在架语义。

## 5. 状态定义（弹窗内）

### 5.1 门店维度 `status`

| 值 | 展示文案 | 含义（产品） |
|----|----------|--------------|
| `success` | 门店：成功 | 该门店维度下发成功 |
| `pending` | 门店：下发中 | 该门店仍在下发中 |
| `failed` | 门店：失败 | 该门店维度下发失败 |

未知值回退为灰底文案显示原始值。

### 5.2 渠道维度 `channelStatuses[].status`

与门店徽标使用同一套颜色映射（成功 / 下发中 / 失败）。用于：

- 向用户展示「同一门店在不同渠道上的差异」；
- 参与 **「重试失败下发」按钮是否显示」**（见 §3.3）。

### 5.3 展示上下架 `shelfStatus`（仅 `status === 'success'`）

| 条件 | 展示标签 | 含义 |
|------|----------|------|
| `shelfStatus !== 'off_shelf'`（含未设置） | **已上架** | 允许点 **下架** |
| `shelfStatus === 'off_shelf'` | **已下架** | 允许 **重新上架**、**删除**（删除需确认） |

**不**对 `pending` / `failed` 门店渲染上下架区（无在架/下架业务）。

## 6. 操作动作与业务逻辑

### 6.1 关闭弹窗

- 点击 **×**、遮罩、或 `Esc`（在无移除确认层时）→ 关闭门店详情；同时关闭移除确认层（若打开）。  
- **不**自动清空 `localStorage` 中筛选键（筛选控件在弹窗内，关闭后下次打开会重置）。

### 6.2 筛选 / 搜索 / 重置

- **纯前端过滤**，不写库；仅改变 `storesListContainer` 内展示的子集。  
- MID 匹配：`String(s.mid).toLowerCase().includes(midQuery)`。

### 6.3 下架（单店）

| 项目 | 说明 |
|------|------|
| **前置** | 当前行 `status === 'success'` 且非已下架。 |
| **动作** | `shelfStatus = 'off_shelf'`，`saveThemes`，刷新弹窗与外层列表卡片。 |
| **结果** | 该行出现「已下架」+「重新上架」「删除」。 |

### 6.4 重新上架（单店）

| 项目 | 说明 |
|------|------|
| **前置** | `status === 'success'` 且 `shelfStatus === 'off_shelf'`。 |
| **动作** | `delete store.shelfStatus`（与「在架」缺省一致），`saveThemes`，刷新。 |

### 6.5 删除（单店，从下发记录移除）

| 项目 | 说明 |
|------|------|
| **前置** | `status === 'success'` 且 **必须已下架**（`shelfStatus === 'off_shelf'`）。 |
| **动作** | 打开 **移除门店** 自定义确认框；确认后从 `distributedStores` 中过滤掉该 `mid`，`saveThemes`，关闭确认框，刷新弹窗与列表。 |
| **取消** | 关闭确认框，不修改数据。 |

### 6.6 重试失败下发

| 项目 | 说明 |
|------|------|
| **入口** | 见 §3.3；`confirm` 文案：「确认重试失败的门店或渠道下发？」。 |
| **范围** | 对当前主题 `distributedStores` **逐条**处理（`map` 不可变引用但就地改对象字段）。 |
| **门店 `failed`** | 随机：`~70%` 变 `success`，`~20%` 变 `pending`，其余保持 `failed`；若变为 `success` 则写入 `successTime`、清除 `failedTime`，且若当前不是下架态则置 `shelfStatus = 'on_shelf'`；若仍为 `failed` 则刷新 `failedTime`；若变为 `pending` 则清除 `failedTime`。 |
| **渠道 `failed`** | 在该门店的 `channelStatuses` 上：每条失败渠道约 `75%` 变 `success`，否则保持 `failed`。 |
| **持久化** | `saveThemes`；再次 `openStoresModal` + 外层 `render()`。 |

**产品说明**：当前为**原型模拟**（随机），对接真实后端时应替换为接口调用、轮询或任务结果回调，并明确幂等与冲突策略。

## 7. 判断逻辑汇总表

| 判断点 | 逻辑 |
|--------|------|
| 是否打开弹窗 | 主题存在且 `distributedStores?.length > 0` |
| 统计数字 | 全部门店计数，不受筛选影响 |
| 是否显示「重试失败下发」 | `failedCount > 0` **或** `hasChannelLevelFailure(stores)` |
| 列表是否展示某门店 | `status` 下拉匹配（空为全部）且 MID 子串匹配 |
| 是否渲染上下架区 | 仅 `store.status === 'success'` |
| 下架是否可点 | 成功且当前非 `off_shelf` |
| 重新上架 / 删除是否可点 | 成功且 `shelfStatus === 'off_shelf'` |
| 删除确认后移除哪一条 | `String(s.mid) !== String(pendingRemoveStoreMid)` |
| 点击操作按钮时定位门店 | `theme.distributedStores.find(s => String(s.mid) === String(data-mid))` |

## 8. 约束条件（产品规则）

1. **仅成功门店**可做下架 / 重新上架 / 删除；失败、下发中门店无此类操作入口。  
2. **删除前必须先下架**（成功门店），避免误删仍在展示在架的记录。  
3. **不支持编辑**门店名称与 MID（无入口、无保存逻辑）。  
4. **筛选与搜索不改变** `distributedStores` 存储，仅影响展示。  
5. **统计条始终反映全部门店**，避免用户误以为筛选后「成功数变少」是数据被改。  
6. 修改门店记录或重试后 **同步刷新外层列表**（与主题卡片上的下发统计一致）。  
7. **重试**使用系统 `confirm`：与「删除屏保」等自定义弹窗体验不一致，若产品统一交互可后续替换。

## 9. 与主题级状态的关系（只读认知）

- 主题卡片上的 **下发状态**（待下发 / 下发中 / 已下发 / …）由 `computeDistributeStatus(distributeTime, distributedStores)` 根据**全部门店 `status`** 等计算，**不读取** `shelfStatus`。  
- 因此：**单店下架/上架不会单独改变**主题级「已下发」等文案；若删除门店导致集合变化，聚合结果可能变化。  
- **生效状态**不在弹窗内计算，但运营需知：**屏保在架/下架**（`effectShelfStatus`）约束的是列表菜单，与弹窗能否打开无关。
