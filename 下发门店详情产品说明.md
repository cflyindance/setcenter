# 关联门店详情 — 产品说明（文档辅助）

> **实现位置**：`kiosk-theme-list.html` 内模态层 `#storesModal` 及相关脚本。  
> **入口文案**：列表卡片 **「查看详情」**（原「下发门店详情」口径已统一为关联门店维度的展示与渠道操作）。  
> **数据归属**：当前主题 `theme.distributedStores`（与 `localStorage` 的 `kiosk_themes` 中该条主题一并持久化）。  
> **上游写入**：门店清单在完成生效时间配置页时由 `effective-time.html` 写入（保存草稿流程）；与页面衔接见 [`屏保新建到完成流程说明.md`](屏保新建到完成流程说明.md)。  
> **主题级状态**：上架状态、生效时间等见 [`屏保状态与操作说明.md`](屏保状态与操作说明.md)。

---

## 1. 产品定位与目标

| 项目 | 说明 |
|------|------|
| **是什么** | 在「屏保主题列表」中，针对**已有关联门店记录**的主题，查看 **门店清单**、每家门店的 **时区 / 门店生效状态**，以及 **各渠道的展示中或未展示**；在允许条件下可对渠道执行 **不展示 / 恢复展示**，并支持 **全渠道** 批量操作。 |
| **不是什么** | 不是重新选择门店的下发向导（那是 `store-select.html`）；**不支持**在弹窗内修改门店名称或 MID；**不展示**门店维度「下发成功 / 下发中 / 失败」统计（当前原型不在此弹窗做下发结果看板）。 |
| **典型用户** | 运营核对门店与渠道展示策略、按渠道或全渠道调整「展示中 / 未展示」。 |

---

## 2. 入口与打开条件

| 条件 | 行为 |
|------|------|
| 列表卡片存在 **「查看详情」** | 仅当该主题已有 `distributedStores` 且 `length > 0` 时展示。 |
| 用户点击 | 调用 `openStoresModal(themeId)`，写入 `currentModalThemeId`，重置 **MID** 与 **生效状态** 筛选，渲染列表；若存在可操作门店则恢复渠道展示操作的 pending 计时器（原型）。 |
| 主题无门店数据 | **不打开**（无提示）。 |

---

## 3. 界面结构

### 3.1 顶栏

| 元素 | 说明 |
|------|------|
| 标题 | 「关联门店详情」 |
| 关闭 | 「×」；点击遮罩关闭；`Esc` 关闭。 |

### 3.2 汇总区（两行）

| 行 | 内容 |
|----|------|
| 第一行 | **总计**：`distributedStores.length` |
| 第二行 | **活动主题时间**：主题统一生效时间文案（优先 `effectTime`；若配置了 `effectStartUtc` / `effectEndUtc` 等可按列表展示逻辑回显）。 |
| 第二行（续） | **门店生效时间**：固定说明为 **「按各门店时区展示」**（同一 UTC 窗口在不同 `store.timezone` 下本地时间不同；未配置时区时默认 `Asia/Shanghai`）。 |

### 3.3 筛选区

| 控件 | 行为 |
|------|------|
| **MID** | 实时按子串、**不区分大小写**匹配 `store.mid`。 |
| **生效状态** | 按 **门店维度** 生效状态过滤；选项随 **主题上架状态** 收敛（见下表）。不合法选项在主题切换时自动回到「全部」。 |
| **重置** | 清空 MID 与生效状态，重绘列表。 |
| **筛选提示** `#storesFilteredHint` | 有筛选或结果条数小于总数时：`当前显示 x / y 家门店`。 |

**生效状态筛选项（由 `getListShelfStatus(theme)` 决定）**

| 主题上架状态（列表口径） | 可选筛选项 |
|--------------------------|------------|
| 待上架 | 全部、待生效 |
| 已上架 | 全部、待生效、生效中、已失效 |
| 已下架 | 全部、已失效 |

> **门店生效状态展示规则**：当主题为「待上架」时，门店侧统一映射为 **待生效**；为「已下架」时统一映射为 **已失效**；仅在 **已上架** 时，才按生效时间窗计算 **待生效 / 生效中 / 已失效**（与列表 `getThemeEffectStatus` 一致）。这样不会出现「未上架却生效中」的组合。

### 3.4 门店卡片（每条）

| 区域 | 内容 |
|------|------|
| 基础信息 | 门店名称、`MID`、**时区**（`store.timezone`，缺省 `Asia/Shanghai`） |
| 门店生效状态 | 徽标：待生效 / 生效中 / 已失效（规则见上） |
| **各关联渠道** | 见 §5 |

## 4. 渠道展示：可操作条件

渠道 **不展示 / 恢复展示**、**全渠道不展示 / 全渠道展示** 仅在同时满足时可用：

- 主题 **上架状态** 为 **已上架**（`getListShelfStatus(theme).text === '已上架'`）；
- 该门店 **生效状态** 为 **生效中**（在已上架前提下由时间窗计算）。

否则整店渠道区为 **只读**：各渠道统一展示 **未展示**，并可有灰色说明（如活动待生效、活动已失效、主题待上架等，由 `getStoreDetailLockedHint` 决定）。

---

## 5. 各关联渠道（展示与操作）

### 5.1 可操作时的单渠道行

| 当前判断（数据 + 展示口径） | 展示文案 | 操作 |
|-----------------------------|----------|------|
| 渠道为「展示中」 | 绿色徽标 **展示中** | **不展示**（写入异步流程，见 §6） |
| 渠道为「未展示」 | 灰色徽标 **未展示** | **恢复展示** |

**说明**：弹窗内 **不展示**「执行中」「执行失败」「重试」等中间态 UI；为保持操作反馈一致，`channelIsOffShelf` 在存在 `displaySync` 时会按 **目标操作方向** 映射展示（例如 pending 隐藏时仍显示为未展示），避免界面出现执行中条。

### 5.2 全渠道批量按钮

在 **§4** 条件满足的前提下，按当前各渠道 **稳定态**（展示中 / 未展示，不含中间态展示）统计：

| 情况 | 展示的按钮 |
|------|------------|
| 全部渠道 **展示中** | 仅 **全渠道不展示** |
| 全部渠道 **未展示** | 仅 **全渠道展示** |
| **混合**（部分展示中、部分未展示） | **全渠道不展示** + **全渠道展示** |

## 6. 数据与异步语义（原型）

- `channelStatuses[]`：`channel`、可选 `shelfStatus: 'off_shelf'`、可选 `displaySync`（pending / failed 等）。  
- 用户点击「不展示 / 恢复展示」仍会写入 `displaySync` 并走原型计时器（成功则更新 `shelfStatus`）；**弹窗内不再展示**执行中、失败与重试按钮。若需排查仍可在存储 JSON 中查看 `displaySync`。  
- 兼容旧数据：门店级 `store.shelfStatus === 'off_shelf'` 仍可能使各渠道表现为未展示。

## 7. 操作动作摘要

| 动作 | 说明 |
|------|------|
| 关闭弹窗 | × / 遮罩 / Esc，不写库。 |
| 筛选 / 搜索 / 重置 | 仅影响展示子集。 |
| 单渠道不展示 / 恢复展示 | 见 §4、§5、事件委托内二次校验可操作门店。 |
| 全渠道批量 | `applyStoreBatchChannelAction`，同上约束。 |

**当前未实现（与旧版文档差异）**

- 弹窗内 **无**「删除门店」「重试失败下发」。  
- **无** 门店维度 success/pending/failed 统计条。
