# 下发门店详情 — 产品说明（文档辅助）

> **实现位置**：`kiosk-theme-list.html` 内模态层 `#storesModal`、移除确认 `#storeRemoveConfirmModal` 及相关脚本。  
> **数据归属**：当前主题 `theme.distributedStores`（与 `kiosk_themes` 中该条主题一并持久化在 `localStorage`）。  
> **上游写入**：门店列表在完成下发流程时由 `effective-time.html` 生成（含模拟下发结果）；与流程衔接见 [`屏保新建到完成流程说明.md`](屏保新建到完成流程说明.md)。  
> **状态语义补充**：主题级下发/生效聚合见 [`屏保状态与操作说明.md`](屏保状态与操作说明.md)。

---

## 1. 产品定位与目标

| 项目 | 说明 |
|------|------|
| **是什么** | 在「屏保主题列表」中，针对**已执行过下发**的主题，查看本次下发涉及的**门店清单**及每家门店的**门店维度结果**、**分渠道结果**，并在规则允许时进行**按渠道的上下架**及**全渠道均已下架后的门店记录删除**。 |
| **不是什么** | 不是重新选择门店的下发向导（那是 `store-select.html`）；**不支持**在弹窗内修改门店名称或 MID。列表页 **不展示** 主题级「下发状态」聚合。 |
| **典型用户** | 运营 / 配置人员：核对下发结果、处理失败重试、按渠道控制已上架/已下架与记录清理。 |

---

## 2. 入口与打开条件

| 条件 | 行为 |
|------|------|
| 列表卡片上存在 **「查看详情」** 按钮 | 仅当该主题已有 `distributedStores` 且 `length > 0` 时展示（无下发记录则无入口）。 |
| 用户点击 **查看详情** | 调用 `openStoresModal(themeId)`。 |
| 主题存在且 `distributedStores` 非空 | 打开弹窗：写入 `currentModalThemeId`，重置筛选项（门店状态 = 全部、MID 为空），统计成功/下发中/失败数量，决定是否展示 **重试失败下发**，再渲染列表。 |
| 主题不存在或无 `distributedStores` | **不打开**弹窗（无提示，静默）。 |

说明：当前实现**不**因主题处于「逻辑已生效且未屏保下架」而禁止打开本弹窗（可查看详情并做允许范围内的门店操作）。

## 3. 界面结构（信息架构）

### 3.1 顶栏

| 元素 | 说明 |
|------|------|
| 标题 | 「下发门店详情」 |
| 关闭 | 「×」；点击遮罩空白处关闭；`Esc` 关闭（若移除确认层已打开则先关确认层）。 |

### 3.2 统计条（全量，不受筛选影响）

| 指标 | 计算逻辑 |
|------|----------|
| **总计** | `distributedStores.length` |
| **成功** | `status === 'success'` 的条数 |
| **下发中** | `status === 'pending'` 的条数 |
| **失败** | `status === 'failed'` 的条数 |

每次打开弹窗或刷新弹窗内数据时按**当前主题全部门店**重算；与下方列表筛选**无关**。

### 3.3 「重试失败下发」按钮

| 显示条件（满足其一即可） | 判断逻辑 |
|--------------------------|----------|
| 存在失败门店 | `failedCount > 0` |
| 存在渠道级失败 | `hasChannelLevelFailure(stores)`：至少一家门店的 `channelStatuses` 中存在 `status === 'failed'` |

否则按钮 **隐藏**。  
点击后打开 **`#retryFailedConfirmModal`** 二次确认（取消 / 确定），确定后执行重试逻辑（见 §6）。

### 3.4 筛选与提示区

| 控件 | 行为 |
|------|------|
| **门店状态** 下拉 | 选项：全部 / 成功 / 下发中 / 失败；变更时重绘列表（仅过滤展示，不改数据）。 |
| **MID** 输入框 | 实时按子串、**不区分大小写**匹配 `store.mid`；空则不过滤 MID。 |
| **重置** | 清空状态筛选与 MID，重绘列表。 |
| 筛选结果提示 `#storesFilteredHint` | 当有筛选条件或结果条数小于总数时显示：`当前显示 x / y 家门店`。 |
| 黄色说明文案 | 固定提示：下发**成功**的渠道可**按渠道**下架/重新上架；该门店下**所有已成功渠道**均已下架后，可**删除该门店**记录；**不支持**在此修改门店名称或 MID。 |

### 3.5 门店列表（每条卡片）

| 区域 | 内容 |
|------|------|
| 左侧主信息 | 门店名称、`MID:`；成功且有 `successTime` 时展示「下发成功: &lt;时间&gt;」；失败且有 `failedTime` 时展示「下发失败: &lt;时间&gt;」；失败且无 `failedTime`（旧数据）时展示「下发失败: —」 |
| 右上徽标 | **门店：** + 门店维度状态（成功 / 下发中 / 失败），颜色与列表内约定一致 |
| **各展示渠道与上下架** | 当 `channelStatuses` 存在且非空时展示；每行：渠道名 + 该渠道同步状态；若该渠道为 **成功**，另附已上架/已下架与 **下架 / 重新上架** 按钮 |
| **删除门店** | 仅当 **门店维度为成功** 且 **每条渠道成功记录均已下架** 时显示「从记录中删除此门店」（见 §5、§6） |

无匹配筛选结果时展示空态文案：「无匹配门店，请调整门店状态或 MID 关键词」。

## 4. 数据模型（单条 `distributedStores[i]`）

以下为原型中与弹窗强相关的字段（非穷尽主题上所有字段）。

| 字段 | 类型（概念） | 含义 |
|------|--------------|------|
| `name` | 字符串 | 门店名称（展示，不可在弹窗内编辑） |
| `mid` | 字符串 | 门店 MID；操作中作为**行内唯一键**匹配（字符串比较） |
| `status` | `success` \| `pending` \| `failed` | **门店维度**下发结果 |
| `successTime` | 字符串（可选） | 门店维度为成功时的展示时间（完成下发或重试成功时写入） |
| `failedTime` | 字符串（可选） | 门店维度为失败时的展示时间（完成下发或重试仍为失败时写入；成功后会清除） |
| `channelStatuses` | `{ channel, status, shelfStatus? }[]`（可选） | 各渠道在该门店上的下发结果；`status` 取值与门店级类似。对 **`status === 'success'`** 的渠道，可选 `shelfStatus: 'off_shelf'` 表示该渠道**已下架**；缺省表示**已上架** |
| `shelfStatus`（门店级） | `'off_shelf'` 或缺省 | **兼容旧数据**：历史上曾用门店级字段表示整店下架；新数据以 **`channelStatuses[].shelfStatus`** 为准。若仍存在门店级 `off_shelf`，则所有 **渠道成功** 行在展示上视为已下架，直至用户在某渠道「重新上架」时清除门店级字段 |

写入时机概要：完成下发时在 `effective-time.html` 内按场景随机生成门店与渠道状态；**不再**写入门店级「整店」上下架占位；重试成功时清除门店级 `shelfStatus` 及各成功渠道上的 `shelfStatus`（恢复为已上架语义）。

## 5. 状态定义（弹窗内）

### 5.1 门店维度 `status`

| 值 | 展示文案 | 含义（产品） |
|----|----------|--------------|
| `success` | 门店：成功 | 该门店维度下发成功 |
| `pending` | 门店：下发中 | 该门店仍在下发中 |
| `failed` | 门店：失败 | 该门店维度下发失败 |

未知值回退为灰底文案显示原始值。

### 5.2 渠道维度 `channelStatuses[].status`

与门店徽标使用同一套颜色映射（成功 / 下发中 / 失败）。用于：

- 向用户展示「同一门店在不同渠道上的差异」；
- 参与 **「重试失败下发」按钮是否显示」**（见 §3.3）。

### 5.3 展示上下架（仅 `channelStatuses[].status === 'success'`）

| 条件（单渠道） | 展示 | 操作 |
|----------------|------|------|
| 该渠道未下架（`channelStatuses[i].shelfStatus !== 'off_shelf'`，且不因旧数据门店级 `store.shelfStatus === 'off_shelf'` 而视为已下架时同上） | **已上架** + **下架** | 写入该条 `shelfStatus = 'off_shelf'` |
| 该渠道已下架 | **已下架** + **重新上架** | `delete channelStatuses[i].shelfStatus`；若存在旧数据 `store.shelfStatus === 'off_shelf'` 则一并 `delete store.shelfStatus` |

**删除整条门店记录**：仅当门店 `status === 'success'` 且 **所有「渠道成功」行**均已下架时，展示「从记录中删除此门店」；确认后从 `distributedStores` 移除该 `mid`。

**不**对 `pending` / `failed` 的渠道行渲染上下架按钮。

## 6. 操作动作与业务逻辑

### 6.1 关闭弹窗

- 点击 **×**、遮罩、或 `Esc`（在无移除确认层时）→ 关闭门店详情；同时关闭移除确认层（若打开）。  
- **不**自动清空 `localStorage` 中筛选键（筛选控件在弹窗内，关闭后下次打开会重置）。

### 6.2 筛选 / 搜索 / 重置

- **纯前端过滤**，不写库；仅改变 `storesListContainer` 内展示的子集。  
- MID 匹配：`String(s.mid).toLowerCase().includes(midQuery)`。

### 6.3 下架（单渠道）

| 项目 | 说明 |
|------|------|
| **前置** | 该门店 `status === 'success'`，且目标渠道在 `channelStatuses` 中 `status === 'success'` 且当前已上架。 |
| **动作** | 对该渠道对象 `shelfStatus = 'off_shelf'`，`saveThemes`，刷新弹窗与外层列表。 |

### 6.4 重新上架（单渠道）

| 项目 | 说明 |
|------|------|
| **前置** | 同上门店成功，且该渠道成功行当前为已下架。 |
| **动作** | `delete` 该渠道对象上的 `shelfStatus`；若存在旧数据 `store.shelfStatus === 'off_shelf'` 则 `delete store.shelfStatus`；`saveThemes`，刷新。 |

### 6.5 删除（单店，从下发记录移除）

| 项目 | 说明 |
|------|------|
| **前置** | `status === 'success'`，且 **每一条** `channelStatuses` 中 `status === 'success'` 的记录均已下架（含旧数据门店级整店下架的等价判断）。 |
| **动作** | 打开 **移除门店** 确认框；确认后从 `distributedStores` 中过滤掉该 `mid`，`saveThemes`，关闭确认框，刷新弹窗与列表。 |
| **取消** | 关闭确认框，不修改数据。 |

### 6.6 重试失败下发

| 项目 | 说明 |
|------|------|
| **入口** | 见 §3.3；弹出 **重试失败下发** 对话框，说明为随机模拟更新失败项后保存。 |
| **范围** | 对当前主题 `distributedStores` **逐条**处理（`map` 不可变引用但就地改对象字段）。 |
| **门店 `failed`** | 随机：`~70%` 变 `success`，`~20%` 变 `pending`，其余保持 `failed`；若变为 `success` 则写入 `successTime`、清除 `failedTime`，并 `delete store.shelfStatus`、清除各成功渠道上的 `shelfStatus`；若仍为 `failed` 则刷新 `failedTime`；若变为 `pending` 则清除 `failedTime`。 |
| **渠道 `failed`** | 在该门店的 `channelStatuses` 上：每条失败渠道约 `75%` 变 `success`，否则保持 `failed`；渠道新变为 `success` 时 `delete` 该条上的 `shelfStatus`。 |
| **持久化** | `saveThemes`；再次 `openStoresModal` + 外层 `render()`。 |

**产品说明**：当前为**原型模拟**（随机），对接真实后端时应替换为接口调用、轮询或任务结果回调，并明确幂等与冲突策略。

## 7. 判断逻辑汇总表

| 判断点 | 逻辑 |
|--------|------|
| 是否打开弹窗 | 主题存在且 `distributedStores?.length > 0` |
| 统计数字 | 全部门店计数，不受筛选影响 |
| 是否显示「重试失败下发」 | `failedCount > 0` **或** `hasChannelLevelFailure(stores)` |
| 列表是否展示某门店 | `status` 下拉匹配（空为全部）且 MID 子串匹配 |
| 是否渲染渠道上下架 | 该门店 `status === 'success'` 且对应渠道 `channelStatuses[i].status === 'success'` |
| 渠道下架是否可点 | 该渠道当前已上架（见 `channelIsOnShelf` 等价：非 `off_shelf` 且无旧数据整店下架） |
| 渠道重新上架是否可点 | 该渠道当前已下架 |
| 「删除此门店」是否展示 | `allSuccessChannelsOffShelf`：门店成功且每条渠道成功记录均已下架 |
| 删除确认后移除哪一条 | `String(s.mid) !== String(pendingRemoveStoreMid)` |
| 点击操作按钮时定位门店 | `theme.distributedStores.find(s => String(s.mid) === String(data-mid))` |

## 8. 约束条件（产品规则）

1. **仅门店维度成功**时展示渠道列表与删除入口；**仅渠道下发成功**的行显示该渠道上下架按钮；失败、下发中门店无删除入口。  
2. **删除门店前**：该门店下 **每一条渠道成功记录** 均须已下架（或旧数据整店下架等价），避免误删仍有已上架渠道的记录。  
3. **不支持编辑**门店名称与 MID（无入口、无保存逻辑）。  
4. **筛选与搜索不改变** `distributedStores` 存储，仅影响展示。  
5. **统计条始终反映全部门店**，避免用户误以为筛选后「成功数变少」是数据被改。  
6. 修改门店记录或重试后 **同步刷新外层列表**（与主题卡片上的下发统计一致）。  
7. **重试**与移除门店、删除屏保一致，使用页面内 **自定义确认框**（`#retryFailedConfirmModal`），支持遮罩点击关闭、`Esc` 关闭（与门店详情弹窗的 Escape 层级：先关重试确认，再关移除确认，最后关详情）。

## 9. 与主题级状态的关系（只读认知）

- 主题列表卡片 **不展示** 主题级「下发状态」；各门店 **成功 / 下发中 / 失败** 仅在 **下发门店详情** 中展示与筛选。  
- 因此：**单渠道下架/上架**不会单独改变主题级「已下发」等文案；若删除门店导致集合变化，聚合结果可能变化。  
- **生效状态**不在弹窗内计算，但运营需知：**屏保在架/下架**（`effectShelfStatus`）约束的是列表菜单，与弹窗能否打开无关。
