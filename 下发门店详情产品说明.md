# 下发门店详情 — 产品说明（文档辅助）

> **实现位置**：`kiosk-theme-list.html` 内模态层 `#storesModal`、移除确认 `#storeRemoveConfirmModal` 及相关脚本。  
> **数据归属**：当前主题 `theme.distributedStores`（与 `kiosk_themes` 中该条主题一并持久化在 `localStorage`）。  
> **上游写入**：门店列表在完成下发流程时由 `effective-time.html` 生成（含模拟下发结果）；与流程衔接见 [`屏保新建到完成流程说明.md`](屏保新建到完成流程说明.md)。  
> **状态语义补充**：主题级下发/生效聚合见 [`屏保状态与操作说明.md`](屏保状态与操作说明.md)。

---

## 1. 产品定位与目标

| 项目 | 说明 |
|------|------|
| **是什么** | 在「屏保主题列表」中，针对**已执行过下发**的主题，查看本次下发涉及的**门店清单**及每家门店的**门店维度结果**、**各展示渠道**；**按渠道的不展示 / 恢复展示**仅当该门店为**下发成功**；**删除门店**任意门店维度状态均可（与渠道展示状态无关）。 |
| **不是什么** | 不是重新选择门店的下发向导（那是 `store-select.html`）；**不支持**在弹窗内修改门店名称或 MID。列表页 **不展示** 主题级「下发状态」聚合。 |
| **典型用户** | 运营 / 配置人员：核对下发结果、处理失败重试、按渠道控制已展示/已不展示与记录清理。 |

---

## 2. 入口与打开条件

| 条件 | 行为 |
|------|------|
| 列表卡片上存在 **「查看详情」** 按钮 | 仅当该主题已有 `distributedStores` 且 `length > 0` 时展示（无下发记录则无入口）。 |
| 用户点击 **查看详情** | 调用 `openStoresModal(themeId)`。 |
| 主题存在且 `distributedStores` 非空 | 打开弹窗：写入 `currentModalThemeId`，重置筛选项（门店状态 = 全部、MID 为空），统计成功/下发中/失败数量，决定是否展示 **重试失败下发**，再渲染列表。 |
| 主题不存在或无 `distributedStores` | **不打开**弹窗（无提示，静默）。 |

说明：当前实现**不**因主题处于「逻辑已生效且未屏保下架」而禁止打开本弹窗（可查看详情并做允许范围内的门店操作）。

## 3. 界面结构（信息架构）

### 3.1 顶栏

| 元素 | 说明 |
|------|------|
| 标题 | 「下发门店详情」 |
| 关闭 | 「×」；点击遮罩空白处关闭；`Esc` 关闭（若移除确认层已打开则先关确认层）。 |

### 3.2 统计条（全量，不受筛选影响）

| 指标 | 计算逻辑 |
|------|----------|
| **总计** | `distributedStores.length` |
| **成功** | `status === 'success'` 的条数 |
| **下发中** | `status === 'pending'` 的条数 |
| **失败** | `status === 'failed'` 的条数 |

每次打开弹窗或刷新弹窗内数据时按**当前主题全部门店**重算；与下方列表筛选**无关**。

### 3.3 「重试失败下发」按钮

| 显示条件（满足其一即可） | 判断逻辑 |
|--------------------------|----------|
| 存在失败门店 | `failedCount > 0` |
| 存在渠道级失败（旧数据） | `hasChannelLevelFailure(stores)`：至少一家门店的 `channelStatuses` 中存在 `status === 'failed'`（新下发不再写入渠道 `status`） |

否则按钮 **隐藏**。  
点击后打开 **`#retryFailedConfirmModal`** 二次确认（取消 / 确定），确定后执行重试逻辑（见 §6）。

### 3.4 筛选与提示区

| 控件 | 行为 |
|------|------|
| **门店状态** 下拉 | 选项：全部 / 成功 / 下发中 / 失败；变更时重绘列表（仅过滤展示，不改数据）。 |
| **MID** 输入框 | 实时按子串、**不区分大小写**匹配 `store.mid`；空则不过滤 MID。 |
| **重置** | 清空状态筛选与 MID，重绘列表。 |
| 筛选结果提示 `#storesFilteredHint` | 当有筛选条件或结果条数小于总数时显示：`当前显示 x / y 家门店`。 |
| 黄色说明文案 | 固定提示：各展示渠道可**按渠道**设为**不展示**或**恢复展示**（**仅当该门店维度为「成功」**）；每家门店均可**删除门店**；**不支持**在此修改门店名称或 MID。 |

### 3.5 门店列表（每条卡片）

| 区域 | 内容 |
|------|------|
| 左侧主信息 | 门店名称、`MID:`；成功且有 `successTime` 时展示「下发成功: &lt;时间&gt;」；失败且有 `failedTime` 时展示「下发失败: &lt;时间&gt;」；失败且无 `failedTime`（旧数据）时展示「下发失败: —」 |
| 右上徽标 | **门店：** + 门店维度状态（成功 / 下发中 / 失败），颜色与列表内约定一致 |
| **各展示渠道** | 当 `channelStatuses` 存在且非空时展示；每行：渠道名 + 状态区。**不展示 / 恢复展示 / 重试** 等可操作项**仅当** `store.status === 'success'`；门店为 **下发中 / 失败** 时只读：在架态文案为 **「展示」**（非「已展示」）、下架态仍为 **「已不展示」**；若有 `displaySync` 则只读说明（无按钮）；**执行中**在云端未收到 Kiosk 回传时可长期保持；不在渠道行展示历史「渠道下发成功/失败」徽标 |
| **删除门店** | 每条门店卡片底部均显示「**删除门店**」按钮；点击后经二次确认从 `distributedStores` 移除该 `mid`（见 §6） |

无匹配筛选结果时展示空态文案：「无匹配门店，请调整门店状态或 MID 关键词」。

## 4. 数据模型（单条 `distributedStores[i]`）

以下为原型中与弹窗强相关的字段（非穷尽主题上所有字段）。

| 字段 | 类型（概念） | 含义 |
|------|--------------|------|
| `name` | 字符串 | 门店名称（展示，不可在弹窗内编辑） |
| `mid` | 字符串 | 门店 MID；操作中作为**行内唯一键**匹配（字符串比较） |
| `status` | `success` \| `pending` \| `failed` | **门店维度**下发结果 |
| `successTime` | 字符串（可选） | 门店维度为成功时的展示时间（完成下发或重试成功时写入） |
| `failedTime` | 字符串（可选） | 门店维度为失败时的展示时间（完成下发或重试仍为失败时写入；成功后会清除） |
| `channelStatuses` | `{ channel, shelfStatus?, displaySync? }[]`（可选） | 各展示渠道；**不再**使用 `status` 表示渠道下发成功/失败（新下发仅含 `channel`）。可选 `shelfStatus: 'off_shelf'` 表示该渠道**不展示**（仅当云端已收到 Kiosk **执行成功**回传后由配置端写入）；缺省表示**已展示**。可选 **`displaySync`**：云端下发展示指令 → Kiosk 执行 → Kiosk 回传执行结果 → 云端更新状态，见下表 |
| `shelfStatus`（门店级） | `'off_shelf'` 或缺省 | **兼容旧数据**：历史上曾用门店级字段表示整店不展示；新数据以 **`channelStatuses[].shelfStatus`** 为准。若仍存在门店级 `off_shelf`，则各渠道行在展示上视为**已不展示**，直至用户在某渠道「恢复展示」时清除门店级字段 |

写入时机概要：完成下发时在 `effective-time.html` 内按场景随机生成**门店维度**状态；`channelStatuses` 仅含所选渠道名（及后续用户操作的 `shelfStatus`）；**不再**写入门店级「整店」占位；重试成功时清除门店级 `shelfStatus` 及各渠道上的 `shelfStatus`（恢复为已展示语义），并清除各渠道 **`displaySync`** 与相关前端计时器。

#### `channelStatuses[i].displaySync`（展示控制异步操作）

| 字段 | 类型（概念） | 含义 |
|------|--------------|------|
| `state` | `'pending'` \| `'failed'` | `pending`：**执行中**（云端已下发指令，等待 Kiosk 执行并回传结果，且等待云端把最终结果同步到配置端）；**不因等待时长自动变为失败**。`failed`：仅表示 **Kiosk 执行未成功**，且**执行失败结果已由云端收到**后，配置端展示为失败（可重试） |
| `action` | `'hide'` \| `'show'` | 与本次操作一致：`hide` = 不展示；`show` = 恢复展示 |
| `startedAt` | 数字（时间戳，可选） | `pending` 时记录发起时刻（原型可用于确定性模拟） |
| `reason` | 字符串（可选） | `failed` 时：`exec_failed` = Kiosk 执行失败且结果已到达云端；历史数据可能存在旧值 `timeout`（已不再作为产品语义） |

产品链路：**云端下发** → **Kiosk 收到并执行**（成功或失败）→ **Kiosk 将执行结果反馈云端** → **云端收到结果**后更新业务状态：成功则配置端结束执行中并写入/清除 `shelfStatus`，失败则配置端展示失败。**若云端一直收不到 Kiosk 的执行结果回传**，配置端**一直保持执行中**（`pending`），不设「超时自动失败」。原型在 `kiosk-theme-list.html` 内用分支模拟：成功回传、失败回传（`exec_failed`）、长期无回传（保持 `pending`）。

## 5. 状态定义（弹窗内）

### 5.1 门店维度 `status`

| 值 | 展示文案 | 含义（产品） |
|----|----------|--------------|
| `success` | 门店：成功 | 该门店维度下发成功 |
| `pending` | 门店：下发中 | 该门店仍在下发中 |
| `failed` | 门店：失败 | 该门店维度下发失败 |

未知值回退为灰底文案显示原始值。

### 5.2 渠道维度（展示控制）

新下发数据中 **`channelStatuses` 不含 `status`**，不在弹窗内区分渠道「下发成功/失败」。旧数据中若仍存在 `channelStatuses[].status === 'failed'`，仅用于 **「重试失败下发」是否显示**（见 §3.3），**不在**渠道行展示该徽标。

### 5.3 展示 / 不展示（按渠道）

| 条件（单渠道） | 展示 | 操作 |
|----------------|------|------|
| `displaySync.state === 'pending'` 且 `action === 'hide'` | **执行中**（Kiosk 已收云端指令，执行「不展示」；等回传与云端状态） | **门店成功时**：无按钮；说明云端未收到回传则一直保持。**门店非成功时**：只读提示，无操作 |
| `displaySync.state === 'pending'` 且 `action === 'show'` | **执行中**（同上，执行「恢复展示」） | 同上 |
| `displaySync.state === 'failed'` | **执行失败**（Kiosk 执行失败且云端已收到失败结果）+ **重试**（仅门店成功时） | 门店成功时：重新进入 `pending` 并再次走链路。**门店非成功时**：只读说明，无重试按钮 |
| 该渠道当前为已展示（无 `off_shelf`，且不因旧数据门店级 `store.shelfStatus === 'off_shelf'` 而视为已不展示），且无上述 `pending`，且 **`store.status === 'success'`** | **已展示** + **不展示** | 写入 `displaySync = { state:'pending', action:'hide', startedAt }`；**云端在收到 Kiosk 执行成功回传后**再写入 `shelfStatus = 'off_shelf'` 并清除 `displaySync` |
| 该渠道当前为已不展示，且无 `pending`，且 **`store.status === 'success'`** | **已不展示** + **恢复展示** | 写入 `displaySync = { state:'pending', action:'show', startedAt }`；**云端在收到 Kiosk 执行成功回传后**再 `delete` 该渠道 `shelfStatus`；若存在旧数据 `store.shelfStatus === 'off_shelf'` 则一并 `delete store.shelfStatus`，并清除 `displaySync` |
| 门店非 `success` 且无 `displaySync` 或仅静态渠道态 | **`failed` / `pending` 门店**：在架只读 **「展示」**（不用「已展示」）；**已不展示**（只读）+ 灰色说明「门店成功后可操作」 | 不写入 `displaySync`；无 **不展示 / 恢复展示** 按钮 |

**删除整条门店记录**：每条门店均展示「**删除门店**」按钮；确认后从 `distributedStores` 移除该 `mid`，与门店维度状态、渠道是否已不展示**无关**；移除前应取消该门店各渠道上挂起的展示控制计时器（原型实现）。

## 6. 操作动作与业务逻辑

### 6.1 关闭弹窗

- 点击 **×**、遮罩、或 `Esc`（在无移除确认层时）→ 关闭门店详情；同时关闭移除确认层（若打开）。  
- **不**自动清空 `localStorage` 中筛选键（筛选控件在弹窗内，关闭后下次打开会重置）。

### 6.2 筛选 / 搜索 / 重置

- **纯前端过滤**，不写库；仅改变 `storesListContainer` 内展示的子集。  
- MID 匹配：`String(s.mid).toLowerCase().includes(midQuery)`。

### 6.3 不展示（单渠道）

| 项目 | 说明 |
|------|------|
| **前置** | **`store.status === 'success'`**；能在列表中定位到该门店与渠道行，且当前为 **已展示**（`channelIsOnShelf`：无 `off_shelf`、无 `displaySync.pending`）。 |
| **动作** | 写入 `displaySync = { state:'pending', action:'hide', startedAt }` 并持久化；界面 **执行中**；**仅当**云端已收到 Kiosk **执行成功**回传并同步后，再写入 `shelfStatus = 'off_shelf'` 并删除 `displaySync`。若 Kiosk **执行失败**且失败结果到达云端，则置 `failed`（`reason: exec_failed`），界面仍保持已展示语义。若云端**始终收不到** Kiosk 回传，则**一直保持** `pending`。 |

### 6.4 恢复展示（单渠道）

| 项目 | 说明 |
|------|------|
| **前置** | **`store.status === 'success'`**；当前该渠道为 **已不展示**（无 `pending`）。 |
| **动作** | 写入 `displaySync = { state:'pending', action:'show', startedAt }` 并持久化；界面 **执行中**；**仅当**云端已收到 Kiosk **执行成功**回传并同步后，再 `delete` 该渠道 `shelfStatus` 及必要时门店级 `shelfStatus`，并删除 `displaySync`。执行失败且结果到云端则 `failed`；无回传则**一直** `pending`。 |

### 6.5 删除（单店，从下发记录移除）

| 项目 | 说明 |
|------|------|
| **前置** | 无（任意门店维度状态均可从下发记录中移除此门店）。 |
| **动作** | 打开 **移除门店** 确认框；确认后从 `distributedStores` 中过滤掉该 `mid`，`saveThemes`，关闭确认框，刷新弹窗与列表。 |
| **取消** | 关闭确认框，不修改数据。 |

### 6.6 重试失败下发

| 项目 | 说明 |
|------|------|
| **入口** | 见 §3.3；弹出 **重试失败下发** 对话框，说明为随机模拟更新失败项后保存。 |
| **范围** | 对当前主题 `distributedStores` **逐条**处理（`map` 不可变引用但就地改对象字段）。 |
| **门店 `failed`** | 随机：`~70%` 变 `success`，`~20%` 变 `pending`，其余保持 `failed`；若变为 `success` 则写入 `successTime`、清除 `failedTime`，并 `delete store.shelfStatus`、清除各成功渠道上的 `shelfStatus` 与 **`displaySync`**、取消渠道计时器；若仍为 `failed` 或变为 `pending`，亦清除各渠道 **`displaySync`** 并取消计时器（门店非成功时不应保留渠道异步操作）。 |
| **渠道 `failed`（旧数据）** | 新逻辑**不再**在重试中改写渠道 `status`；若旧数据存在渠道级 `failed`，重试逻辑仍以门店维度为主（渠道行以展示控制为准，不含成功/失败徽标）。 |
| **持久化** | `saveThemes`；再次 `openStoresModal` + 外层 `render()`。 |

**产品说明**：当前为**原型模拟**（随机），对接真实后端时应替换为接口调用、轮询或任务结果回调，并明确幂等与冲突策略。

## 7. 判断逻辑汇总表

| 判断点 | 逻辑 |
|--------|------|
| 是否打开弹窗 | 主题存在且 `distributedStores?.length > 0` |
| 统计数字 | 全部门店计数，不受筛选影响 |
| 是否显示「重试失败下发」 | `failedCount > 0` **或** `hasChannelLevelFailure(stores)` |
| 列表是否展示某门店 | `status` 下拉匹配（空为全部）且 MID 子串匹配 |
| 是否渲染渠道展示控制 | `channelStatuses` 非空即渲染各行（无渠道级成功/失败徽标） |
| 「不展示」是否可点 | **`store.status === 'success'`** 且该渠道当前为已展示且无 `displaySync.pending`（`channelIsOnShelf`） |
| 「恢复展示」是否可点 | **`store.status === 'success'`** 且该渠道当前为已不展示且无 `displaySync.pending`（`channelIsOffShelf`） |
| 渠道 **重试** 是否可点 | **`store.status === 'success'`** 且 `displaySync.state === 'failed'` |
| 是否恢复渠道 `displaySync` 原型计时器 | `openStoresModal` 内仅对 **`store.status === 'success'`** 的门店下仍为 `pending` 的渠道调用 `armChannelDisplayOpTimers` |
| 「执行中」是否展示 | 存在 `displaySync.state === 'pending'` |
| 「执行中」何时结束 | **仅**在云端已同步 **Kiosk 执行成功**（写入/清除 `shelfStatus` 并删 `displaySync`）或 **执行失败且结果已到云端**（`failed` + `reason: exec_failed`）时结束；**不因等待时长自动结束** |
| 「执行中」若云端一直收不到 Kiosk 回传 | **一直保持** `pending`（不设超时自动失败） |
| 「执行失败 / 重试」是否展示 | 存在 `displaySync.state === 'failed'`（语义为 Kiosk 执行失败且云端已收到失败结果） |
| 「删除门店」按钮是否展示 | 每条门店均展示（与 `status`、渠道展示状态无关） |
| 删除确认后移除哪一条 | `String(s.mid) !== String(pendingRemoveStoreMid)` |
| 点击操作按钮时定位门店 | `theme.distributedStores.find(s => String(s.mid) === String(data-mid))` |

## 8. 约束条件（产品规则）

1. **`channelStatuses` 非空** 时展示各渠道行；**不展示 / 恢复展示 / 重试** 仅当 **`store.status === 'success'`**；**门店 `failed` / `pending`** 时渠道在架只读文案为 **「展示」**（非「已展示」）。**门店成功**时：展示 **已展示 / 已不展示** 与上述操作；异步时展示 **执行中**（可长期保持）或 **执行失败 + 重试**（仅 Kiosk 执行失败且结果已到达云端）；**不**在渠道行展示历史「渠道下发成功/失败」类同步徽标。  
2. **删除门店**：任意门店均可从下发记录中移除；删除前经二次确认。  
3. **不支持编辑**门店名称与 MID（无入口、无保存逻辑）。  
4. **筛选与搜索不改变** `distributedStores` 存储，仅影响展示。  
5. **统计条始终反映全部门店**，避免用户误以为筛选后「成功数变少」是数据被改。  
6. 修改门店记录或重试后 **同步刷新外层列表**（与主题卡片上的下发统计一致）。  
7. **重试**与移除门店、删除屏保一致，使用页面内 **自定义确认框**（`#retryFailedConfirmModal`），支持遮罩点击关闭、`Esc` 关闭（与门店详情弹窗的 Escape 层级：先关重试确认，再关移除确认，最后关详情）。

## 9. 与主题级状态的关系（只读认知）

- 主题列表卡片 **不展示** 主题级「下发状态」；各门店 **成功 / 下发中 / 失败** 仅在 **下发门店详情** 中展示与筛选。  
- 因此：**单渠道不展示/恢复展示**不会单独改变主题级「已下发」等文案；若删除门店导致集合变化，聚合结果可能变化。  
- **生效状态**不在弹窗内计算，但运营需知：**屏保在架/下架**（`effectShelfStatus`）约束的是列表菜单，与弹窗能否打开无关。
