# 活动上下架状态与门店生效状态组合展示建议

## 背景与前提

在屏保场景中存在两层状态（与 `kiosk-theme-list.html` 中 `getListShelfStatus` / `getStoreEffectStatus` 一致）：

- **上架状态（主题级、列表展示）**：`待上架` / `已上架` / `已下架`  
  - 由 `getListShelfStatus(theme)` 计算，**不等同于**仅看 `publishStatus` 字段（见下文「口径要点」）。
- **门店生效状态（详情内门店行）**：`待生效` / `生效中` / `已失效`  
  - 由 `getStoreEffectStatus(theme, store)` 计算；**当前原型里状态判定使用主题级生效时间窗**（`effectStartUtc` / `effectEndUtc` / `effectTime` 等），`store` 仅参与时区等展示逻辑，**同一主题下各门店徽标文案一致**。

核心业务前提（与实现一致）：

> 只有上架状态为 **`已上架`** 时，门店状态才可能出现 **`生效中`**。  
> 上架状态为 **`待上架`** 或展示为 **`已下架`** 时，门店侧不会出现 **`生效中`**。

---

## 口径要点（与当前代码对齐）

### 「上架状态」三态如何得来

| 列表展示 | 条件（摘要） |
|----------|----------------|
| **待上架** | `publishStatus === 'draft'`（含用户点击 **下架** 后回到草稿的情形） |
| **已上架** | `publishStatus === 'published'` 且主题级生效状态 **不是** `已失效` |
| **已下架** | `publishStatus === 'published'` 且主题级生效状态为 **`已失效`**（**生效时间窗已结束**） |

因此：

- **用户主动「下架」** → 数据变为草稿 → 列表为 **`待上架`**，**不是**「已下架」。
- 列表中的 **「已下架」** 特指 **仍保持已发布（published）但活动整体已因时间结束而失效** 的展示口径，与运营口语里的「下架」需区分。

主题级生效状态由 `getThemeEffectStatus` / `parseStoredThemeEffectRange` + `computeEffectStatusByRange` 计算（UTC 时间窗或 `effectTime` 解析）。

### 「门店生效状态」如何受上架状态约束

实现见 `getStoreEffectStatus`：

- 上架状态为 **待上架** → 门店统一 **`待生效`**（不再按时间窗算出生效中）。
- 上架状态为 **已下架**（published + 已失效）→ 门店统一 **`已失效`**。
- 上架状态为 **已上架** → 与主题级相同时间窗 → **`待生效` / `生效中` / `已失效`**。

渠道是否可操作：`isStoreDetailChannelUiOperable` → 需 **`已上架`** 且门店侧为 **`生效中`**（与 `getStoreEffectStatus` 一致）。

---

## 设计原则

- 采用「**主从状态**」思路：
  - **主状态**：上架状态（决定门店状态可用空间、筛选项收敛）。
  - **从状态**：门店生效状态（在主状态允许范围内，由**主题级**时间窗计算）。
- 避免出现语义冲突组合：
  - `待上架 + 生效中`
  - `已下架（时间到期展示）+ 生效中`
- **注意**：`待上架 + 手动下架后的草稿` 在数据层仍是草稿，列表不会显示「已下架」三态中的「已下架」，避免与「时间到期已下架」混淆。

---

## 状态组合规则（与实现一致）

### 1）上架状态 = 待上架

- 门店状态统一为：**`待生效`**
- 门店状态不可出现：**`生效中`**
- 渠道操作：**不可**（默认未展示；锁定提示见 `getStoreDetailLockedHint` →「主题待上架」等）

### 2）上架状态 = 已上架

- 门店状态按 **主题级**生效时间窗计算：**`待生效` / `生效中` / `已失效`**
- 渠道操作：仅当门店侧为 **`生效中`** 时可操作

### 3）上架状态 = 已下架（仅：published 且时间窗已结束）

- 门店状态统一为：**`已失效`**
- 门店状态不可出现：**`生效中`**
- 渠道操作：**不可**

---

## 页面展示建议

### 列表页（主题级）

- 继续以上架状态（`getListShelfStatus`）为主展示；**不**与门店级逐条混排，避免误解。
- 若产品需要「主题级生效概况」，应与 `getThemeEffectStatus` 一致，并标明是 **主题时间窗**，不是某一家门店独立规则（除非后续数据模型支持门店级时间窗）。

### 关联门店详情（门店级）

- 展示：**门店生效状态**徽标、`getStoreEffectFilterOptionsByTheme` 收敛后的筛选、活动/门店时间说明（时区格式化见 `formatThemeEffectRangeForStore`）。
- 渠道可操作性：**上架状态 = 已上架** 且 **门店生效状态 = 生效中**（`isStoreDetailChannelUiOperable`）。

---

## 门店生效状态筛选（动态约束）

与 `getStoreEffectFilterOptionsByTheme` + `renderStoresEffectStatusFilter` 一致：

| 上架状态（列表） | 筛选项 |
|------------------|--------|
| 待上架 | `全部` / `待生效` |
| 已上架 | `全部` / `待生效` / `生效中` / `已失效` |
| 已下架（时间到期） | `全部` / `已失效` |

交互：若重建选项后当前选中值不在新列表中，**自动回退为「全部」**（`storesEffectStatusFilterEl.value = ''`）。

---

## 推荐结论

1. **`生效中`** 应理解为：**上架状态为「已上架」前提下**，由主题级生效时间窗决定的子状态；不是与上架平级的独立维度。
2. **列表「已下架」** 与 **操作「下架」** 含义不同：前者为 **发布时间未撤、仅因到期在 UI 上显示为已下架**；后者将主题打回草稿，列表为 **待上架**。
3. 当前原型中门店行「生效状态」与主题时间窗一致；若未来要做 **按门店不同生效区间**，需单独扩展数据结构与 `getStoreEffectStatus` 计算逻辑，再同步本文。
